How  Locking  Prevents  Double  Spending  Imagine  two  people  somehow  try  to  withdraw  from  the  same  account  at  the  exact  
same
 
millisecond.
 
Without
 
protection:
 ·
        
Both  read  the  balance  as  $1,000  ·
        
Both  think  there  is  enough  money  ·
        
Both  withdraw  $800  ·
        
Result:  $1,600  withdrawn  from  a  $1,000  account.  Money  was  created  out  of  
thin
 
air.
 Our  solution:  FOR  UPDATE  locks.  When  a  stored  procedure  reads  the  balance,  it  
also
 
LOCKS
 
that
 
row.
 
Any
 
other
 
procedure
 
that
 
tries
 
to
 
read
 
the
 
same
 
row
 
has
 
to
 
WAIT
 
until
 
the
 
first
 
one
 
finishes.
 This  means:  3.
      
Procedure  A  reads  balance  ($1,000),  locks  the  row  4.
      
Procedure  B  tries  to  read,  but  has  to  wait  5.
      
Procedure  A  withdraws  $800,  balance  becomes  $200,  COMMIT,  releases  
lock
 6.
      
Procedure  B  finally  reads  balance  ($200),  sees  it  is  not  enough,  rejects  Only  one  succeeds.  This  is  correct.  
How  We  Prevent  Deadlocks  in  Transfers  A  deadlock  is  when  two  operations  wait  for  each  other  forever:  ·
        
Transfer  A  locks  Account  1,  then  tries  to  lock  Account  2  ·
        
Transfer  B  locks  Account  2,  then  tries  to  lock  Account  1  ·
        
Both  are  stuck  waiting.  Forever.  Our  solution:  Always  lock  the  account  with  the  smaller  ID  first.  This  way,  both  
transfers
 
try
 
to
 
lock
 
the
 
same
 
account
 
first.
 
One
 
gets
 
the
 
lock,
 
the
 
other
 
waits.
 
No
 
deadlock.

Key  Terms  Glossary  
Term  Plain  English  
Table  A  spreadsheet  in  the  database.  Each  one  stores  one  type  of  thing.  
Row  /  Record  One  entry  in  a  table  (like  one  row  in  a  spreadsheet).  
Column  /  Field  One  piece  of  information  (like  one  column  in  a  spreadsheet).  
Foreign  Key  (FK)  A  link  from  one  table  to  another.  Like  saying  "this  account  belongs  to  customer  #5."  
Primary  Key  (PK)  The  unique  ID  for  each  row.  No  two  rows  can  have  the  same  one.  
Stored  Procedure  A  saved  set  of  instructions  in  the  database.  The  website  says  "run  this"  and  the  database  does  the  work.  
View  A  saved  query.  Instead  of  writing  the  same  complex  lookup  every  time,  we  save  it  and  give  it  a  name.  
Transaction  (DB)  A  group  of  database  operations  that  must  all  succeed  or  all  fail.  Not  the  same  as  a  banking  transaction.  
COMMIT  Save  everything  we  just  did.  
ROLLBACK  Undo  everything  we  just  did.  
FOR  UPDATE  A  lock  that  says  "nobody  else  can  touch  this  row  until  I  am  done  with  it."  
DECIMAL  A  number  type  that  handles  money  exactly.  FLOAT  can  have  tiny  rounding  errors.  
JWT  (Token)  A  digital  pass.  After  you  log  in,  the  server  gives  you  one.  You  show  it  on  every  request.  
bcrypt  A  one-way  encryption  for  passwords.  You  can  turn  a  password  INTO  a  hash,  but  not  back.  
InnoDB  The  engine  inside  MySQL  that  supports  transactions  and  row  locking.  
API  Route  A  URL  on  the  backend  that  the  frontend  calls  to  get  or  send  data.  


=== PAGE 14 ===
Atomic  All-or-nothing.  Either  every  step  succeeds,  or  none  of  them  do.  
Audit  Trail  A  complete  record  of  what  happened  and  when,  so  you  can  trace  any  event.  
Deadlock  Two  operations  stuck  waiting  for  each  other  forever.  We  prevent  this  with  lock  ordering.  
Defense  in  Depth  Multiple  layers  of  security.  If  one  layer  fails,  the  next  one  catches  it.  
Normalization  Organizing  data  so  nothing  is  stored  in  two  places.  
Seed  Data  Pre-loaded  sample  data  so  the  system  is  not  empty  when  you  first  start  it.
